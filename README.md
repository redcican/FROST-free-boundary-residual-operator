# FROST — Free-boundary Residual Operator

Source code for the paper *Learning and differentiating free-boundary equilibria for forward
simulation and inverse design* (FROST). FROST represents the solution of a free-boundary problem as
a joint fixed point of the physical field and a level set on a fixed background grid, solves and
trains that fixed point by implicit differentiation (in the manner of a deep equilibrium model), and
differentiates the same equilibrium for inverse design.

This repository ships **only the source code and the data-generation scripts** — no trained
checkpoints, predictions, figures or other results. Each script writes its own outputs to a local
`results/` folder when it is run.

## Repository layout

One folder per benchmark. Each benchmark has a forward scenario (`C1/`) and, where it carries a
design task, an inverse-design scenario (`C2_inverse/`):

| benchmark        | paper            | `C1/` (forward)                                                                  | `C2_inverse/` (inverse design)                                  |
|------------------|------------------|----------------------------------------------------------------------------------|-----------------------------------------------------------------|
| `obstacle/`      | Fig. 2 / SI S4.6 | steady contact operator (+ DEQ, few-shot, local-stencil and diffeomorphism studies) | design the obstacle to a target contact set; active trust-region loop |
| `tumour_merge/`  | Fig. 3           | time-dependent coalescing-front operator (forward only)                          | —                                                               |
| `channel/`       | Fig. 4 / SI S4.4 | turbulent-channel concentration / pressure operator                              | baffle-density design with the physics-residual trust signal    |
| `stefan/`        | Fig. 5 / SI S4.1 | solidification forward operator                                                  | control of a topology-changing solidification front             |

Each benchmark root holds its data-generation script (`gen_*.py`), dataset-visualisation scripts
(`viz_*.py`) and a `README.md`. The shared spectral backbone (`fno.py`) and the
`train_*` / `design_*` / `control_*` entry scripts live in `C1/` and `C2_inverse/`. Per-benchmark
and per-scenario `README.md` files give the exact commands and hyperparameters.

## Data

The datasets are **not** bundled with the code. They are regenerated by running the `gen_*.py`
script in each benchmark folder, which writes that benchmark's `*.npy` dataset and a small
`*_summary.json`, e.g.

    python obstacle/gen_obstacle.py        # -> obstacle/obstacle.npy
    python stefan/gen_stefan.py            # -> stefan/stefan.npy
    python tumour_merge/gen_tumour_merge.py
    python channel/gen_channel.py

The channel benchmark additionally rasterises the public Neural-Topology-Optimization (NTO) CFD
dataset into its training arrays via `channel/C1/gen_channel_train.py`. The exact arrays used in the
paper are also deposited on Zenodo (DOI assigned on publication).

A few scripts reuse the published **FBNO** dataset (Long et al., Zenodo
[10.5281/zenodo.15779011](https://doi.org/10.5281/zenodo.15779011)), which they expect in a sibling
`FBNO/` folder next to this package: the primary tumour generator `tumour_merge/gen_tumour_merge.py`
(and `tumour_merge/viz_tumour_merge_3d.py`) build on FBNO's tumour domain, and
`stefan/calibrate_from_fbno.py` calibrates against FBNO's 1-D Stefan solutions. These comparisons are
not required to reproduce the FROST results; a self-contained tumour generator,
`tumour_merge/gen_tumour_merge_disks_v1.py`, is provided as an alternative that needs no external data.

## Environment

Python 3.11 with PyTorch. Create the environment with

    conda env create -f environment.yml
    conda activate frost

or install the core dependencies directly:

    torch>=2.0
    numpy
    scipy
    matplotlib
    scikit-image
    shapely
    pandas

A CUDA-enabled PyTorch build is recommended for training; the scripts fall back to CPU.

## Running

For each benchmark: generate the data, train the forward operator (`C1/`), then optionally run the
inverse design (`C2_inverse/`). For the obstacle problem, for example:

    python obstacle/gen_obstacle.py                   # data        -> obstacle/obstacle.npy
    python obstacle/C1/train_c1_obstacle.py           # forward op  -> obstacle/C1/results/
    python obstacle/C2_inverse/design_c2_obstacle.py  # design      -> obstacle/C2_inverse/results/

See each benchmark's `README.md` (and the `C1/` / `C2_inverse/` READMEs) for the complete set of
commands, the auxiliary studies, and the visualisation scripts.
